# JuhRadial MX v0.2.12 - Animation & UI Improvements Plan

**Date:** 2026-02-27
**Status:** Draft / Planning
**Focus:** Radial wheel animations, visual polish, media state feedback

---

## Current State (v0.2.11)

The radial wheel currently has **one animation** total:
- **Window fade-in/out**: 180ms `QPropertyAnimation` on `windowOpacity` with `OutCubic` easing
  - File: `overlay/juhradial-overlay.py` lines 131-134

Everything else is instant:
- Slice hover highlight: immediate color swap (no transition)
- Submenu appearance: instant pop (no transition)
- Icon states: static (play icon never changes to pause)
- Center zone: no animations
- Slice selection: no visual confirmation feedback

Logitech Options+ has smooth, fluid micro-animations throughout their radial wheel - hover transitions, submenu "droplet" pop-outs, and subtle scaling effects. v0.2.12 should close this gap.

## Improvement 1: Media Play/Pause State Icon

### What
The top slice (Play/Pause) currently always shows a static "play" triangle icon regardless of whether media is actually playing or paused. It should dynamically reflect the real playback state.

### Behavior
- **When stopped/paused**: Show play triangle icon (current behavior)
- **When playing**: Show pause icon (two vertical bars)
- **State query**: `playerctl status` returns `Playing`, `Paused`, or `Stopped`
- **Refresh**: Query state each time the radial menu opens (`on_show`)
- **Per-slice**: Each radial wheel instance queries independently on show

### Implementation Approach

**File: `overlay/overlay_actions.py`**
- Add `get_media_state()` function that runs `playerctl status` (subprocess, non-blocking)
- Cache result in module-level variable `MEDIA_PLAYING = False`
- Call on each `load_actions_from_config()` (already called every `on_show`)

**File: `overlay/overlay_painting.py`**
- In `_draw_icon()`, when icon type is `play_pause`:
  - If `MEDIA_PLAYING` is `True`: draw two vertical bars (pause icon)
  - If `MEDIA_PLAYING` is `False`: draw play triangle (current)
- New icon drawing: two rectangles at ~35% and ~65% horizontal offset

**Pause Icon Geometry:**
```
Bar 1: rect(cx - 6, cy - 8, 4, 16)
Bar 2: rect(cx + 2, cy - 8, 4, 16)
```

### Edge Cases
- `playerctl` not installed: default to play icon, never crash
- No active player: default to play icon
- Multiple players: `playerctl status` uses the most recent active player
- Subprocess timeout: 200ms max, fall back to cached state

### Risk: LOW
- Read-only query, no state mutation
- Fallback to current behavior on any error
- No impact on positioning or rendering pipeline

## Improvement 2: Slice Hover Transition Animation

### What
Currently when hovering over a slice, the highlight appears instantly (hard cut). This should be a smooth transition with eased opacity/brightness.

### Behavior
- **Hover enter**: Highlight fades in over ~120ms with `OutCubic` easing
- **Hover exit**: Highlight fades out over ~80ms with `InCubic` easing (faster exit = snappier feel)
- **Continuous**: If moving between slices, the old slice fades out while the new one fades in simultaneously

### Implementation Approach

**File: `overlay/juhradial-overlay.py`**
- Add per-slice animation state: `self.slice_highlights = [0.0] * 8` (0.0 = no highlight, 1.0 = full highlight)
- Add `QTimer` for animation ticks (~16ms interval, 60fps) - reuse existing `_poll_timer` or create dedicated `_anim_timer`
- On slice change: set target for new slice to 1.0, old slice to 0.0
- Each tick: lerp current values toward target with easing factor

**File: `overlay/overlay_painting.py`**
- Modify `_draw_slice()` to accept `highlight_progress` float (0.0-1.0) instead of boolean `is_highlighted`
- **Vector mode** (lines 310-327): Interpolate alpha from 80 (unhovered) to 45+white overlay based on progress
- **3D mode** (`_draw_3d_slice_highlight`): Multiply highlight fill/border alpha by progress value

**Animation Math:**
```python
# Per-tick update (called every 16ms)
HOVER_IN_SPEED = 0.15   # reaches 1.0 in ~7 ticks = ~112ms
HOVER_OUT_SPEED = 0.20  # reaches 0.0 in ~5 ticks = ~80ms

for i in range(8):
    target = 1.0 if i == self.highlighted_slice else 0.0
    if self.slice_highlights[i] < target:
        self.slice_highlights[i] = min(1.0, self.slice_highlights[i] + HOVER_IN_SPEED)
    elif self.slice_highlights[i] > target:
        self.slice_highlights[i] = max(0.0, self.slice_highlights[i] - HOVER_OUT_SPEED)
```

### Visual Effect (Vector Mode)
```python
# Interpolated highlight
alpha = int(80 + (45 - 80) * highlight_progress)  # 80 -> 45
white_overlay = int(45 * highlight_progress)  # 0 -> 45
```

### Visual Effect (3D Mode)
```python
# Multiply existing theme alpha by progress
fill_alpha = int(theme_highlight_fill[3] * highlight_progress)
border_alpha = int(theme_highlight_border[3] * highlight_progress)
```

### Risk: MEDIUM
- Animation timer must be properly cleaned up on menu hide to prevent CPU leak (Issue #3 regression)
- Must not affect slice detection math (angles/distances unchanged)
- Timer start/stop must be tied to menu visibility
- Test: verify `_anim_timer.stop()` is called in `on_hide()` and `_close_menu()`

## Improvement 3: Submenu "Droplet" Pop-Out Animation

### What
Currently when hovering over a submenu slice (AI, Easy-Switch), the submenu items appear instantly. They should animate outward from the parent slice with a smooth "droplet" effect - items emerge from the wheel edge and settle into position.

### Behavior
- **Open**: Items start at the menu ring outer edge and animate outward to `SUBMENU_RADIUS` (195px) over ~200ms
- **Easing**: `OutBack` (slight overshoot then settle) for a "droplet falling" feel
- **Stagger**: Each item starts ~30ms after the previous for a cascade effect
- **Scale**: Items start at 0.6x scale and grow to 1.0x during the animation
- **Close**: Items collapse back inward over ~120ms with `InCubic` easing (faster retract)

### Implementation Approach

**File: `overlay/juhradial-overlay.py`**
- Add submenu animation state:
  ```python
  self.submenu_progress = 0.0      # 0.0 = hidden, 1.0 = fully shown
  self.submenu_animating = False
  self.submenu_direction = "in"     # "in" = opening, "out" = closing
  ```
- On submenu open: set `submenu_direction = "in"`, start animation
- On submenu close: set `submenu_direction = "out"`, animate then clear

**File: `overlay/overlay_painting.py`**
- Modify `_draw_submenu()` to use `submenu_progress`:

```python
# Per-item stagger
STAGGER_MS = 30
for i, item in enumerate(submenu):
    item_progress = clamp(submenu_progress - (i * 0.1), 0.0, 1.0)

    # OutBack easing for position
    eased = out_back(item_progress)

    # Interpolate radius: start at ring edge, end at submenu position
    animated_radius = MENU_RADIUS + (SUBMENU_RADIUS - MENU_RADIUS) * eased

    # Interpolate scale: 0.6 -> 1.0
    scale = 0.6 + 0.4 * item_progress

    # Interpolate opacity: 0 -> 255
    opacity = int(255 * min(1.0, item_progress * 2))  # fade in faster than movement
```

**OutBack Easing Function:**
```python
def out_back(t, overshoot=1.4):
    """Slight overshoot then settle - creates 'droplet' feel"""
    t = t - 1
    return t * t * ((overshoot + 1) * t + overshoot) + 1
```

### Visual Sequence
1. Frame 0: Items at wheel edge, scale 0.6, opacity 0
2. Frame 4 (~64ms): Items 60% distance, scale 0.84, opacity 255
3. Frame 8 (~128ms): Items overshoot slightly past target
4. Frame 12 (~192ms): Items settle at final position, scale 1.0

### Risk: MEDIUM
- Must not interfere with submenu hit detection (`_get_subitem_at_position`)
- Hit detection should use final positions even during animation (prevents mis-clicks)
- Timer cleanup critical on menu close
- Test: rapid open/close/open cycle must not leave orphaned animations
- Test: submenu still works in both hold mode and toggle mode

### Behavior
- **Hover enter**: Icon scales from 1.0x to 1.15x over ~100ms with `OutCubic`
- **Hover exit**: Icon scales from 1.15x to 1.0x over ~80ms

### Implementation Approach

**File: `overlay/overlay_painting.py`**
- In `_draw_icon()` and related icon methods:

```python
# New (animated):

```

- Icon opacity boost also interpolated:
```python
# Current: +40 to RGB channels on hover
# New: +int(40 * highlight_progress) to RGB channels
brightness_boost = int(40 * highlight_progress)
```

### Risk: LOW
- Reuses Improvement 2's animation system, no additional timers
- Visual-only change, no geometry impact

### What
When the radial menu appears, the center circle should have a subtle "pulse" effect - a brief scale/glow that settles, drawing the eye to the center.

### Behavior
- **On show**: Center zone scales from 0.9x to 1.05x to 1.0x over ~300ms
- **Easing**: `OutElastic` with low amplitude (subtle bounce)
- **Glow**: Brief brightness pulse on the center border

### Implementation Approach

**File: `overlay/juhradial-overlay.py`**
- Add `self.center_pulse = 0.0` (0.0 = start, 1.0 = settled)
- Animate in sync with the existing window fade-in
- Drive from same animation timer as Improvement 2

**File: `overlay/overlay_painting.py`**
- In `_draw_center()`:
```python
# Scale the center zone
pulse_scale = 0.9 + 0.1 * out_elastic(center_pulse_progress)

# Apply transform
p.save()
p.translate(cx, cy)
p.scale(pulse_scale, pulse_scale)
p.translate(-cx, -cy)
# ... draw center as normal ...
p.restore()
```

### Risk: LOW
- QPainter transform is local (save/restore), won't affect other drawing
- Purely cosmetic, no interaction impact

## Improvement 6: Slice Selection Flash

### What
When a slice is selected (action executed), there's currently no visual confirmation. Add a brief flash/highlight on the selected slice before the menu closes.

### Behavior
- **On select**: Selected slice flashes bright white (alpha 120) for ~80ms
- **Then**: Menu closes with normal fade-out
- **Duration**: Total ~80ms flash + 180ms fade = 260ms from click to gone
- **Skip if**: Action opens a submenu (submenu has its own animation)

### Implementation Approach

**File: `overlay/juhradial-overlay.py`**
- In `_execute_action()`: Before hiding menu, set `self.flash_slice = index`
- Start a `QTimer.singleShot(80, self._post_flash_hide)` to delay the hide
- During flash: `self.update()` triggers repaint with flash overlay

**File: `overlay/overlay_painting.py`**
- In `paintEvent`, after drawing slices:
```python
if self.flash_slice >= 0:
    # Draw bright overlay on flashed slice
    path = self._get_slice_path(flash_slice)
    flash_color = QColor(255, 255, 255, 120)
    p.setBrush(QBrush(flash_color))
    p.setPen(Qt.PenStyle.NoPen)
    p.drawPath(path)
```

### Risk: LOW
- 80ms delay before close is imperceptible as lag
- singleShot timer auto-cleans up
- Visual only, no interaction change

## Animation System Architecture

All improvements share a single animation framework to keep things clean and avoid timer proliferation.

### Shared Animation Timer

```python
# In RadialMenu.__init__()
self._anim_timer = QTimer()
self._anim_timer.setInterval(16)  # ~60fps
self._anim_timer.timeout.connect(self._tick_animations)

# Animation state
self.slice_highlights = [0.0] * 8           # Improvement 2, 4
self.submenu_progress = 0.0                  # Improvement 3
self.subitem_highlights = [0.0] * 8          # Improvement 8
self.center_pulse = 0.0                      # Improvement 5
self.menu_scale = 1.0                        # Improvement 7
self.flash_slice = -1                        # Improvement 6
```

### Timer Lifecycle
- **Start**: On `on_show()` after fade-in begins
- **Stop**: On menu fully hidden (`_close_menu()`) - CRITICAL for CPU (Issue #3)
- **Tick**: Update all animation values, call `self.update()` only if any value changed (dirty flag)

### Easing Functions

```python
# New file: overlay/overlay_easing.py (or add to overlay_constants.py)

def out_cubic(t):
    t -= 1
    return t * t * t + 1

def in_cubic(t):
    return t * t * t

def out_back(t, s=1.4):
    t -= 1
    return t * t * ((s + 1) * t + s) + 1

def out_elastic(t, amplitude=0.3):
    if t == 0 or t == 1:
        return t
    p = 0.3
    s = p / 4
    return pow(2, -10 * t) * math.sin((t - s) * (2 * math.pi) / p) + 1
```

## Performance Budget

**Target:** `paintEvent` must complete in under 8ms (leaves room for 60fps with overhead)

**Current baseline:** ~3-5ms per paint (vector mode), ~2-4ms (3D mode - just image blit + overlays)

**Expected additions:**
- Per-slice highlight interpolation: +0.1ms
- Submenu animation math: +0.1ms
- Extra gradient draws (glow): +0.5ms
- Scale transforms: +0.1ms
- **Total estimated:** +0.8ms = ~4-6ms per paint

**Monitoring:** Add optional timing to `paintEvent` during development:
```python
if os.environ.get("JUHRADIAL_PERF"):
    start = time.perf_counter_ns()
    # ... paint ...
    elapsed_ms = (time.perf_counter_ns() - start) / 1_000_000
    if elapsed_ms > 8:
        print(f"SLOW PAINT: {elapsed_ms:.1f}ms")
```

## Summary

v0.2.12 transforms the radial wheel from "functional but static" to "fluid and polished" through 8 targeted animation improvements. The changes are purely visual/cosmetic with zero impact on core functionality, built entirely on PyQt6's existing animation primitives, and architected around a single shared animation timer to prevent the CPU issues that were fixed in earlier versions.

The play/pause state icon adds practical value beyond aesthetics - users can glance at the wheel to see media state without opening a player.

All improvements are independently implementable and testable, so they can be merged incrementally during the sprint.
